1. Prerequisites

Before we can get started, we need to do a bit of configuration outside and inside the WSL environment to enable the rest of the guide

1.1. Enable the Windows Feature

[!IMPORTANT] It seems as if you don’t longer need to manually enable the Windows Feature manually/yourself, as this comment suggests.

I’ll still leave the following info intact, because we all know what beautiful, reliable and stable pieces of software Microsoft Windows and the Microsoft Store are _{sarcasm off} …

You might want to enable the Windows Subsystem for Linux Windows feature. You can either try opening your Windows Search and searching for Turn Windows features on or off, or you can dig your way through the Control Panel.

[!TIP] For WSLg to work properly, you need to make sure you’re running the Windows 11, or the minimum required version of Windows 10 (Version 2004 or higher / Build 19041 or higher).

1.2. Installing / Updating WSL

[!NOTE] Microsoft recommends installing the Distro from the Microsoft store, but also specifies the commands below.

In my experience, depending on your installation, or whether your Microsoft Store has shot itself in the foot yet or not, the installation is a little incoherent.

When in doubt, use the official Microsoft installation documentation.

To install WSL from a cmd or PowerShell with elevated privileges run wsl --install -d Ubuntu, or if you’ve already installed WSL in the past, also from an elevated prompt, run wsl --update.

1.2.1. Just WorksTM?

[!IMPORTANT] This should only be neccessary if your Windows is locked down, the Microsoft Store is disabled, or similar.

As it turns out, if you’re in a pretty restricting corporate setting, the Windows environment is somewhat … botched. The only way I was able to install WSL successfully is by using the manual installation method.

This includes

  1. Downloading the latest WSL release (the .msixbundle)
  2. Installing it by running
Add-AppxPackage Microsoft.WSL_<version>_<arch>.msixbundle
  1. Proceeding with the Installing ArchWSL step.

[!NOTE] If wsl --update also doesn’t work, in addition to wsl --install [...], you might need to repeat the aforementioned process for updating WSL.

1.3. Installing ArchWSL

[!NOTE] When in doubt, use the ArchWSL documentation.

For me, the simplest procedure was to

  1. Download the latest Arch.zip file from the release page
  2. Extract it to a folder where your user has unrestricted read and write permissions, so ideally stay away from Program Files, etc.
  3. Run the included Arch.exe

  4. Change the root password 

    passwd

  5. Add the wheel group to the allowed sudo users 

    echo "%wheel ALL=(ALL) ALL" > /etc/sudoers.d/wheel

  6. Add a new user and add it to the wheel group 

    useradd -m -G wheel -s /bin/bash <username>

  7. Change the new user’s password 

    passwd <username>

  8. Exit the WSL environment 

    exit

  9. Set the new user as the default one 

    Arch.exe config --default-user <username>

  10. Rebooting or restarting LxssManager with 

    net stop lxssmanager

    and 

    net start lxssmanager

  11. After that reenter WSL with 

    Arch.exe

  12. Setup the pacman keyring 

    sudo pacman-key --init
sudo pacman-key --populate
sudo pacman -Sy archlinux-keyring

  13. Update the base system 

    sudo pacman -Su

[!TIP] You can update your mirrorlist with your nearest mirrors.

  1. It is best practice to backup your original mirrorlist.
sudo cp /etc/pacman.d/mirrorlist /etc/pacman.d/mirrorlist.old
  1. Generate a mirrorlist with the 5 best mirrors and save it to /etc/pacman.d/mirrorlist
curl -s "https://archlinux.org/mirrorlist/?country=DE&protocol=http&protocol=https&ip_version=4&use_mirror_status=on"  | sed -e 's/^#Server/Server/' -e '/^#/d' | rankmirrors -n 5 - | sudo tee /etc/pacman.d/mirrorlist

1.3.1. Just WorksTM v2

[!IMPORTANT] This should only be neccessary if you’re behind a (corporate) http proxy

As I’m sitting behind a corporate http proxy, I had no access to the internet. This can be confirmed by running 

curl https://archlinux.org

To make WSL proxy-aware, I needed to

  1. Populate my ~/.bashrc with 

    export http_proxy=http://<hostname>:<port>
export https_proxy=$http_proxy
export ftp_proxy=$http_proxy

  2. Source it with 

    source ~/.bashrc

  3. Allow sudo to pass these environment variables through by populating /etc/sudoers.d/proxy with 

    Defaults env_keep += "http_proxy https_proxy ftp_proxy"

Rerunning the curl command should now produce a response and pacman get updated and upgraded 

sudo pacman -Syyuu

      st)
